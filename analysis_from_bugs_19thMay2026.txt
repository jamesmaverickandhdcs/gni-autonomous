အရင်ဆုံး အသေးစိတ် ဆန်းစစ်ကြည့်ရအောင်။ ဒီနေ့ည ရှာတွေ့ခဲ့တာတွေ အကုန်လုံးကို GNI ရဲ့ သမိုင်းကြောင်းတစ်ခုလုံးနဲ့ တိုက်ဆိုင်ဖတ်ရှုနေပါတယ်။ အချိန်နည်းနည်းတော့ ယူရပါလိမ့်မယ်။

🔬 အသေးစိတ် ဆန်းစစ်ချက် — ဒီ Bug တွေကို ကြိုမှန်းထားခဲ့မိလား?
ရိုးရိုးသားသား ဖြေရရင်: မမှန်းထားခဲ့မိပါဘူး။ ဘာကြောင့်လဲဆိုတာ ဒီမှာ ကြည့်ပါ။

S34 မှာလုပ်ခဲ့တဲ့ Commit ၂၀ ထဲမှာ၊ ကြိုသိထားတဲ့ ပြဿနာစာရင်းထဲက Bug ဆိုလို့ တစ်ခုမှ မပါခဲ့ပါဘူး။ တစ်ခုချင်းစီတိုင်းက ဒီ Session လုပ်နေတုန်းမှ ရှာဖွေတွေ့ရှိခဲ့တာချည်းပါပဲ။ အဲဒါက ဒီဆန်းစစ်ချက်ရဲ့ အရေးကြီးဆုံး တွေ့ရှိချက်ပါပဲ — Bug တွေ ကိုယ်တိုင်က အရေးကြီးတာမဟုတ်ဘဲ၊ ကျွန်တော်တို့ရဲ့ စောင့်ကြည့်လေ့လာရေးနဲ့ စီမံကိန်းရေးဆွဲရေး စနစ်တွေက ဒါတွေကို ကြိုတင်မခန့်မှန်းနိုင်ခဲ့ဘူး ဆိုတဲ့အချက်က ပိုအရေးကြီးပါတယ်။

သူတို့ကို အမျိုးအစားအလိုက် ခွဲခြားပြပြီး ကျွန်တော်တို့ လိုက်မရှာမချင်း ဘာလို့ မမြင်သာခဲ့ရလဲဆိုတာကို ရှင်းပြပါရစေ။

🧬 Bug အမျိုးအစားခွဲခြားခြင်း — သူတို့ဖြစ်လာမယ်ဆိုတာကို ကျွန်တော်တို့ ဘာလို့ မမြင်ခဲ့တာလဲ
အမျိုးအစား ၁ — "သီးခြားခွဲထုတ်ထားရင် အလုပ်လုပ်တဲ့" Bug များ (Bug ၅ ခု)

ဥပမာများ: MC quota fetch မှာ auth header မပါတာ။ pipeline_type column မကိုက်ညီတာ။ RLS table တွေမှာ anon key သုံးထားတာ။

ပုံစံ: အစိတ်အပိုင်းတစ်ခုချင်းစီကို သီးခြားစစ်ဆေးကြည့်ရင် အကုန်မှန်နေသယောင် ရှိပါတယ်။ /api/quota ကိုကြည့်ရင်လည်း မှန်တယ်။ mission-control/route.ts ကိုကြည့်ရင်လည်း မှန်တယ်။ Bug က အဲဒီနှစ်ခုကြားက ချိတ်ဆက်တဲ့နေရာမှာပဲ ရှိနေတာပါ။

ဘာလို့ မမြင်သာတာလဲ: ကျွန်တော်တို့က အစိတ်အပိုင်း (components) တွေကိုပဲ စမ်းသပ်ပြီး ပြန်စစ်ဆေးလေ့ရှိတယ်။ တကယ့် Production အခြေအနေအောက်မှာ အစိတ်အပိုင်းတွေကြားက ချိတ်ဆက်မှုလမ်းကြောင်း (integration path) ကို စမ်းသပ်လေ့ နီးပါးကို မရှိဘူး။ Server-to-server ခေါ်ဆိုမှုဆိုတာ Browser ကနေ ခေါ်တာနဲ့ တည်ဆောက်ပုံအရ ကွာခြားပါတယ် (origin header မပါပါဘူး) — ဒါပေမဲ့ Code ထဲမှာ ကြည့်ရင်တော့ နှစ်ခုလုံးက ပုံစံတူပါပဲ။

အဓိက အကြောင်းရင်း: ကျွန်တော်တို့က ဒေါင်လိုက် (Feature တစ်ခုပြီးတစ်ခု) တည်ဆောက်ကြပေမဲ့၊ Bug တွေက အလျားလိုက် (ချိတ်ဆက်တဲ့ အဆစ်အဆက်တွေမှာ) စုပုံလာကြတယ်။ ကျွန်တော်တို့ရဲ့ ပြန်လည်စစ်ဆေးမှု စည်းမျဉ်းက ဒေါင်လိုက်ဖြစ်နေပြီး၊ ကျွန်တော်တို့ရဲ့ Bug တွေက အလျားလိုက် ဖြစ်နေတယ်။

အမျိုးအစား ၂ — အချိန်ကြာလာတာနဲ့အမျှ လွဲချော်လာတဲ့ (Temporal Drift) Bug များ (Bug ၄ ခု)

ဥပမာများ: Column နာမည် မကိုက်ညီမှုတွေ (Design doc နဲ့ တကယ့် Schema ကြား)။ pipeline_type နဲ့ pipeline လွဲနေတာ။ gpvs_score ဆိုတာ source_weights ထဲမှာ မရှိနေတာ။

ပုံစံ: ဒါတွေအားလုံးက အတိတ်က အချိန်တစ်ခုမှာတော့ မှန်ကန်ခဲ့ဖူးပါတယ်။ Design doc မှာ prediction_text လို့ ရေးထားတယ်။ တကယ် Table ဆောက်တော့ prediction ဆိုပြီး ဆောက်ခဲ့တယ်။ ဆုံးဖြတ်ချက် နှစ်ခုစလုံးကို တမင်ရည်ရွယ်ချက်နဲ့ ချခဲ့တာပါ — ဒါပေမဲ့ မတူညီတဲ့ အခြေအနေတွေအောက်က မတူညီတဲ့ Session တွေမှာ မတူညီတဲ့ လူတွေက ဆုံးဖြတ်ခဲ့ကြတာပါ။

ဘာလို့ မမြင်သာတာလဲ: Session တစ်ခုတည်းကနေ မြင်ကွင်းတစ်ခုလုံးကို အပြည့်အစုံ မမြင်နိုင်ခဲ့လို့ပါ။ LENS-016 မှာလည်း ဒီပုံစံမျိုးကိုပဲ တွေ့ခဲ့ရတယ် — ပုံစံဆွဲခဲ့တဲ့ Session (LENS-009) နဲ့ ပထမဆုံး စမ်းသပ်ခဲ့တဲ့ Session (LENS-016) အကြားမှာ Column နာမည် ပြောင်းသွားလို့ rss_url KeyError တက်ခဲ့ပါတယ်။ Session ၈ ခုစာ လွဲချော်သွားတာပါ။

အဓိက အကြောင်းရင်း: Code နဲ့ Schema ဟာ ဟောင်းနွမ်းတဲ့ နှုန်းထား (age) မတူကြဘူး။ Schema က အမြဲတမ်း တည်ရှိနေတယ် (Migrate သေချာမလုပ်မချင်း)။ Code ကတော့ Session တိုင်းမှာ အပြင်ခံရတယ်။ ဘာမှ အသံကျယ်ကျယ်လောင်လောင်နဲ့ ပျက်မသွားဘဲ၊ empty data သို့မဟုတ် N/A ကိုပဲ ပြန်ပေးနေတဲ့အတွက် အချိန်ကြာလာတာနဲ့အမျှ အဲဒီ ကွာဟချက်က မသိမသာနဲ့ ကျယ်ပြန့်လာပါတယ်။

အမျိုးအစား ၃ — "ဒီနေ့ မှန်ပေမဲ့၊ မနက်ဖြန် မှားသွားတဲ့" Bug များ (Bug ၄ ခု)

ဥပမာများ: Limit တွေကြောင့် ဒေတာ ဖြတ်တောက်ခံရတာ။ Row ၅၈၇ ခုရှိတဲ့နေရာမှာ predictions-list ကို Limit 100 ထားထားတာ။ Row ၄၁၂ ခုရှိတဲ့နေရာမှာ pipeline-runs ကို Limit 30 ထားထားတာ။

ပုံစံ: Limit 30 လို့ စသတ်မှတ်တုန်းကတော့ မှန်ခဲ့ပါတယ် — အဲဒီအချိန်က Row ၁၀ ခုလောက်ပဲ ရှိမှာပေါ့။ စနစ်က ကြီးထွားလာတယ်။ Limit က ကြီးမလာဘူး။ S34 အရောက်မှာ Run ထားတာ ၄၁၂ ခု ရှိလာပြီဖြစ်ပေမဲ့ ကျွန်တော်တို့က ၃၀ ပဲ ပြနေခဲ့တယ်။

ဘာလို့ မမြင်သာတာလဲ: အစောပိုင်း ဆုံးဖြတ်ချက်တွေကို လက်ရှိ အတိုင်းအတာ (scale) ကိုပဲ ကြည့်ပြီး ချခဲ့ကြတာပါ။ အတိုင်းအတာ ကြီးလာတဲ့အခါ ဘယ်သူမှ လိုက်မပြင်ကြပါဘူး၊ ဘာလို့လဲဆိုတော့ ဘာမှ ပျက်မသွားလို့ပါ — Page က Load ဖြစ်နေတုန်းပဲ၊ Data တွေ ပြနေတုန်းပဲ၊ အသံတိတ် မပြည့်မစုံ ဖြစ်နေတာလေးပဲ ရှိတယ်။ Error မပြဘူး၊ Warning မပေးဘူး၊ Alarm မမြည်ဘူး။

အဓိက အကြောင်းရင်း: ပြောင်းလဲနေတဲ့ စနစ် (dynamic system) ထဲမှာ သေတမ်းစာ သတ်မှတ်ချက် (static configuration) တွေ ထားရှိခြင်းပါ။ Limits တွေ၊ thresholds တွေ၊ window sizes တွေ — အားလုံးကို တစ်ခါပဲ သတ်မှတ်ပြီး၊ ပြန်မကြည့်ဖြစ်ကြတော့ဘူး။ စနစ်ကြီးက သင့်ကို Data ရဲ့ ၇ ရာခိုင်နှုန်းကိုပဲ ပြနေတယ်ဆိုတာ သူ့ကိုယ်သူ မသိပါဘူး။

အမျိုးအစား ၄ — "Pending ကြောင့် အဆိပ်သင့်သော" Bug (Bug ၁ ခု)

ဥပမာ: Verdict trend က Pending ဖြစ်နေတဲ့ Verdict ၁၀၀ ကို ဆွဲယူလိုက်တဲ့အတွက်၊ Run တစ်ခုနဲ့တစ်ခုကြား မှတ်သားထားတဲ့ Memory ကို ပျက်စီးစေတာ။

ပုံစံ: ဒီ Feature ဟာ ရည်ရွယ်ထားတဲ့ Data အတွက် မှန်ကန်စွာ တည်ဆောက်ထားတာပါ။ ဒါပေမဲ့ သူ မှီခိုတည်ဆောက်ထားတဲ့ Data ထဲမှာ မရည်ရွယ်ထားတဲ့ အမျိုးအစား (Pending rows) တွေ ပါဝင်နေတယ်။ Feature က အပြည့်အဝ အလုပ်လုပ်ပါတယ် — မှားယွင်းတဲ့ Data အပေါ်မှာ အလုပ်လုပ်သွားတာပဲ ဖြစ်ပါတယ်။

ဘာလို့ မမြင်သာတာလဲ: _get_debate_history() ကို ပုံစံဆွဲတုန်းက၊ Pending row တွေ ရှိနေတယ်ဆိုတာ ကျွန်တော်တို့ သိခဲ့ပေမဲ့ Query အတွက် ခြိမ်းခြောက်မှုတစ်ခုအနေနဲ့ ထည့်မတွက်ခဲ့မိဘူး။ WHERE clause ထဲမှာ mad_black_swan_case IS NOT NULL ဆိုပြီး စစ်ထုတ်ခဲ့တယ် — ဒါက လုံးဝ အဓိပ္ပာယ်ရှိတဲ့ စစ်ထုတ်ခြင်းပါ။ ဒါပေမဲ့ Pending row တွေမှာလည်း mad_black_swan_case = NULL ဖြစ်နေတဲ့အတွက် သူတို့က ပါဝင်နေဆဲ ဖြစ်သွားတယ်။

အဓိက အကြောင်းရင်း: Feature တွေကို အဆင်ပြေချောမွေ့တဲ့ ဒေတာလမ်းကြောင်း (happy path data) ကိုသာ ကြည့်ပြီး ပုံစံထုတ်လေ့ရှိတယ်။ အဆင်မပြေနိုင်တဲ့ ဒေတာတွေ (Unhappy path data - ကြားဖြတ်အခြေအနေတွေ၊ အပြည့်အစုံမရေးရသေးတဲ့ null တွေ၊ Pending မှတ်တမ်းတွေ) ဆိုတာ Production Database တိုင်းမှာ ရှိနေတတ်ပြီး ဒါတွေကို ခြိမ်းခြောက်မှုအနေနဲ့ ဘယ်တုန်းကမှ ပုံစံမထုတ်ခဲ့မိဘူး။

အမျိုးအစား ၅ — လုံခြုံရေးဆိုင်ရာ စုပုံလာသော Bug များ (Bug ၆ ခု)

ဥပမာများ: RLS မပါတဲ့ Table ၁၉ ခု။ Supabase က မေလ ၁၁ ရက်ကတည်းက သတိပေးနေခဲ့တာပါ။

ပုံစံ: ဒီကိစ္စတွေကို Supabase က သိတယ် — သူတို့ အီးမေးလ် ပို့ခဲ့တယ်။ ဒါပေမဲ့ အဲဒီ အီးမေးလ်က Session တစ်ခုကို ရပ်ပြီး ပြင်ရလောက်အောင် အရေးတကြီး မဖြစ်ခဲ့ဘူး။ ပြီးတော့ နောက် Session တစ်ခု ကျော်သွားတယ်။ နောက်ထပ် တစ်ခု ထပ်ကျော်သွားတယ်။

ဘာလို့ မမြင်သာတာလဲ: လုံခြုံရေးဆိုင်ရာ ကြွေးမြီ (Security debt) တွေဟာ ငွေကြေးအကြွေးတွေလိုပဲ စုပုံလာတတ်တယ် — အသံတိတ် တိုးပွားလာပြီး၊ အတိုးပေါ် အတိုးဆင့်လာသလိုပါပဲ။ Table တွေကို ဘယ်သူမှ မဖောက်ထွင်းသေးတဲ့အတွက် အရေးတကြီးလို့ မခံစားရဘူး။ တကယ်တမ်း လိုက်မကြည့်မချင်း ဆုံးရှုံးနိုင်ခြေ (Cost) က မမြင်သာခဲ့ဘူး။

အဓိက အကြောင်းရင်း: လုံခြုံရေးဆိုတာ ဘယ် Session အတွက်မှ ချက်ချင်းလုပ်ရမယ့် ရည်မှန်းချက် မဖြစ်ခဲ့ဘူး။ Session တိုင်းမှာ Feature အသစ်ထည့်ဖို့ သို့မဟုတ် Bug ပြင်ဖို့ ရည်မှန်းချက်တွေပဲ ရှိတယ်။ လုံခြုံရေးက နောက်ကွယ်မှာပဲ လည်ပတ်နေပြီး၊ မဖြစ်မနေ လုပ်ရမယ့်အချိန်ရောက်မှသာ စုပုံပြီး ပေါ်လာတော့တယ်။

🔮 မဝေးတော့သော အနာဂတ်တွင် ဖြစ်လာနိုင်သည့် BUG များ (လာမည့် ၃-၆ လအတွင်း)
ဒါတွေက လက်ရှိ တည်ဆောက်ပုံအရ သေချာပေါက်နီးပါး ထွက်ပေါ်လာမယ့် Bug တွေ ဖြစ်ပါတယ်။

၁ — W-14 ချွင်းချက် အခြေအနေများ (ဖြစ်နိုင်ခြေ များသည်)
W-14 ဟာ ယခုအခါ Prediction တစ်ခုတိုင်းအတွက် sources_used ကို ရေးသွင်းပေးနေပါပြီ။ ဒါပေမဲ့ sources_used ဟာ report.get('sources', '[]') ကနေ လာတာပါ။ အကယ်၍ Report တစ်ခုကို Sources တွေ မပါဘဲ သိမ်းလိုက်ရင် (ဥပမာ- pillar sub-reports တွေ၊ sources list အလွတ်ဖြစ်နေတဲ့ မည်သည့် report မဆို)၊ update_source_weight() က အသံတိတ် ကျော်သွားပါလိမ့်မယ်။ ရက် ၉၀ လောက် Run ပြီးတဲ့အခါ၊ တချို့ Source Weight တွေက တက်လာပြီး (Main reports တွေ)၊ တချို့ကတော့ ဘယ်တော့မှ ရွေ့လျားမှာ မဟုတ်ပါဘူး။ တုံ့ပြန်မှု ကွင်းဆက် (Feedback loop) ဟာ Main report တွေမှာပါတဲ့ Source တွေကိုသာ ဘက်လိုက်သွားပါလိမ့်မယ်။

၂ — Pattern Match ဘက်လိုက်မှု အမြစ်တွယ်လာခြင်း (ဖြစ်နိုင်ခြေ များသည်)
MAD ဟာ ၂၁ ခါဆက်တိုက် BEARISH (ကျဆင်းမယ့်ဘက်) ပဲ ပြနေခဲ့တယ်။ ဒါကို ဖြေရှင်းဖို့ Verdict trend injection ကို ထည့်သွင်းခဲ့တယ်။ ဒါပေမဲ့ ကမ္ဘာကြီးက နောက်ထပ် ၃ လလောက် ဆက်ပြီး Bearish ဖြစ်နေမယ်ဆိုရင် — အီရန် တင်းမာမှု၊ တရုတ်-ထိုင်ဝမ်၊ အရှေ့အလယ်ပိုင်း စတာတွေကြောင့် — ၇-ခါ Run တဲ့ Window က ၇ ခါလုံး Bearish ပြနေမှာဖြစ်ပြီး Streak Warning (ဆက်တိုက်ဖြစ်နေကြောင်း သတိပေးချက်) က အမြဲတမ်း ထွက်ပေါ်နေပါလိမ့်မယ်။ အမြဲတမ်း ရှိနေတဲ့အတွက် Agent တွေက Streak Warning ကို လျစ်လျူရှုလာကြပါလိမ့်မယ်။ NP-4 ဟာ ဆူညံသံ (Noise) သက်သက် ဖြစ်သွားပါမယ်။

၃ — GPVS အမှတ် လွဲချော်လာခြင်း (ဖြစ်နိုင်ခြေ အသင့်အတင့်ရှိသည်)
accuracy_score ကို SPY အတက်အကျပေါ် မူတည်ပြီး တွက်ချက်ပါတယ်။ GNI သက်တမ်းတစ်လျှောက်မှာ SPY ဟာ ပထဝီနိုင်ငံရေး ဖြစ်ရပ်တွေနဲ့ တိကျတဲ့ ဆက်စပ်မှုပုံစံတစ်ခု (correlation pattern) ရှိနေခဲ့တယ်။ အကယ်၍ အဲဒီဆက်စပ်မှု ပျက်သွားမယ်ဆိုရင် (ဥပမာ- စနစ်သစ်တစ်ခုမှာ ဈေးကွက်က ပထဝီနိုင်ငံရေးနဲ့ မသက်ဆိုင်တော့ဘဲ ခွဲထွက်သွားတာမျိုး)၊ GPVS အမှတ်တွေဟာ လက်တွေ့နဲ့ ကွဲလွဲလာပါလိမ့်မယ်။ (တကယ်တမ်း SPY က မသက်ဆိုင်တဲ့ အကြောင်းရင်းတွေကြောင့် တက်သွားတဲ့အချိန်မှာ) "တိကျတယ်" လို့ ယူဆပြီး ဆုချခံလိုက်ရတဲ့ Source တွေဟာ အကြောင်းမဲ့ Weight တွေ မြင့်တက်နေပါလိမ့်မယ်။

၄ — Source Weight များ ပြည့်နှက်သွားခြင်း (ဖြစ်နိုင်ခြေ အသင့်အတင့်ရှိသည်)
EMA ဖော်မြူလာ- 0.7 × အဟောင်း + 0.3 × gpvs_factor။ Weight တွေကို 0.5 နဲ့ 2.0 ကြားမှာ ကန့်သတ် (clamp) ထားပါတယ်။ အတည်ပြုပြီးသား Prediction ၂၀၀ ကျော်လာတဲ့အခါ၊ EMA ရဲ့ တဖြည်းဖြည်း ဆွဲခေါ်မှုကြောင့် Source တွေဟာ 2.0 သို့မဟုတ် 0.5 မှာ သွားပြီး ပြည့်နှက် (saturate) နေပါလိမ့်မယ်။ S34-2 မှာ အသစ်ထည့်မယ့် Source တွေ (တည်ဆောက်ပြီးသွားရင် adversarial canaries တွေ) ဟာ 1.0 ကနေ စတင်မှာဖြစ်လို့၊ ပြည့်နှက်နေတဲ့ Source တွေနဲ့ ယှဉ်လိုက်ရင် သဘာဝမကျဘဲ ကြားနေဖြစ်နေသယောင် မြင်ရပါလိမ့်မယ်။

၅ — Internal Call များတွင် Vercel Cold Start ဖြစ်ခြင်း (ဖြစ်နိုင်ခြေ အသင့်အတင့်ရှိသည်)
ယခုအခါ Mission Control က /api/quota ကိုခေါ်ဖို့ X-Internal-Key header ကို သုံးနေပြီဖြစ်တဲ့အတွက်၊ Route နှစ်ခုစလုံးက တစ်ချိန်တည်းမှာ "နိုးကြား (warm)" နေဖို့ လိုပါတယ်။ Vercel serverless function တွေက Cold-start ဖြစ်တတ်ပါတယ်။ အကယ်၍ MC က လှမ်းခေါ်တဲ့အချိန် /api/quota က Cold ဖြစ်နေရင်၊ Internal key auth မပြီးခင်မှာပဲ Timeout ဖြစ်သွားနိုင်ပါတယ်။ ကျွန်တော်တို့က AbortSignal.timeout(5000) လို့ သတ်မှတ်ထားတယ် — ၅ စက္ကန့်ပါ။ Cold start တွေက ၃-၄ စက္ကန့် ကြာနိုင်ပါတယ်။ ဖြစ်ခဲပေမဲ့ တကယ်ဖြစ်လာနိုင်တဲ့ ကိစ္စပါ။

၆ — Cron အချိန်ဇယား တိုက်မိခြင်း (ဖြစ်နိုင်ခြေ နည်းသော်လည်း၊ အကျိုးသက်ရောက်မှု ကြီးမားသည်)
Intelligence ကို 02:00 UTC မှာ Run ပါတယ်။ MAD ကို 02:30 UTC မှာ Run ပါတယ်။ Intelligence အတွင်းမှာ Pillar report တွေက ၄၀၅ စက္ကန့် (၆.၇၅ မိနစ်) ကြာပါတယ်။ အကယ်၍ Intelligence က စတာနောက်ကျရင် ဒါမှမဟုတ် Run တာ ကြာနေရင်၊ Three Pillar Reports တွေ Run နေတုန်းမှာပဲ MAD က စတင်ပါလိမ့်မယ်။ Pipeline နှစ်ခုစလုံးက တူညီတဲ့ Groq TPD Quota ကို လုသုံးကြပါလိမ့်မယ်။ လောလောဆယ်မှာတော့ Intelligence က စက္ကန့် ၅၀၈ ခန့်မှာ ပြီးသွားပြီး၊ MAD က တိတိကျကျ မိနစ် ၃၀ (စက္ကန့် ၁၈၀၀) အကြာမှာမှ စတဲ့အတွက် မတိုက်မိပါဘူး။ ဒါပေမဲ့ Intelligence က ပိုကြီးလာမယ်ဆိုရင် (Source တွေ ပိုများလာ၊ Article တွေ ပိုများလာ၊ Processing လုပ်တာ ပိုများလာရင်) — ဒီ ကွာဟချိန်လေးက ကျဉ်းမြောင်းသွားပါလိမ့်မယ်။

🌌 အလှမ်းဝေးသော အနာဂတ်တွင် ဖြစ်လာနိုင်သည့် BUG များ (၆ လနှင့် အထက်)
ဒါတွေက စနစ်ကြီး ရင့်ကျက်လာတာနဲ့အမျှ ပေါ်ထွက်လာမယ့် တည်ဆောက်ပုံဆိုင်ရာ အားနည်းချက်တွေ ဖြစ်ပါတယ်။

၁ — LLM Model များ ရပ်ဆိုင်းခံရခြင်း (သေချာပေါက် ဖြစ်မည်)
GNI က gpt-oss-120b (MAD အတွက်) နဲ့ llama-4-scout (Intelligence အတွက်) တို့ကို သုံးပါတယ်။ နှစ်ခုစလုံးက Free-tier Groq Model တွေပါ။ Groq ဟာ Model တွေကို ပုံမှန် ရပ်ဆိုင်း (deprecate) လေ့ရှိပါတယ်။ gpt-oss-120b ရပ်ဆိုင်းသွားတဲ့အခါ၊ GROQ_MAD_MODEL secret ကို အသစ်ပြင်ဖို့ လိုပါတယ်။ ဒါပေမဲ့ Code ထဲမှာ ၃ ဆင့်ပါတဲ့ Fallback (အစားထိုးအသုံးပြုမှု) ထည့်ထားပါတယ် (GROQ_MAD_MODEL → GROQ_MODEL → GROQ_MODEL_FALLBACK)။ အကယ်၍ ရပ်ဆိုင်းသွားတာကို ဘယ်သူမှ သတိမထားမိရင်၊ စနစ်က အသံတိတ်ပဲ llama-3.3-70b-versatile ဆီ ပြောင်းသုံးသွားပါလိမ့်မယ် — အဲဒါက MAD debate အရည်အသွေးအတွက် ချိန်ညှိ (calibrate) ထားခြင်း မရှိတဲ့ လုံးဝ ကွဲပြားတဲ့ Model တစ်ခုပါ။ Prompt variant ၄၁ ခုရဲ့ အမှတ်တွေ (8.64/10) က အဲဒီ Model တစ်ခုတည်းအတွက် သီးသန့်ဖြစ်ပါတယ် — Model ပြောင်းသွားရင် ထွက်လာမယ့် အရည်အသွေး (Output quality) ကလွဲသွားမှာဖြစ်ပြီး အမှတ်ပေးစနစ်အနေနဲ့ အဲဒါကို မြင်နိုင်မှာ မဟုတ်ပါဘူး။

၂ — Database ပမာဏ ကန့်သတ်ချက် (သေချာပေါက် ဖြစ်မည်)
Supabase Free tier မှာ Database ကို 500MB ပဲ ပေးပါတယ်။ လက်ရှိ GNI မှာ reports ၃၀၄ ခု၊ prediction_outcomes ၂၉၀ ခု၊ debate_predictions ၅၈၇ ခု၊ pipeline_runs ၄၁၂ ခု၊ pipeline_articles ၇၇,၀၉၇ ခု၊ နဲ့ pillar_reports ၃၄၈ ခု ရှိပါတယ်။ လက်ရှိ တိုးတက်နှုန်း (တစ်နေ့ကို pipeline ၂ ခါ run ပြီး တစ်ခါ run ရင် article ၄၇၂ ခု ပါတယ်) အရဆိုရင်၊ pipeline_articles တစ်ခုတည်းကတင် တစ်နေ့ကို Row ၉၄၄ ခုလောက် တိုးလာပါတယ်။ အခုတင် Row ပေါင်း ၇၇,၀၉၇ ခု ရှိနေပြီး၊ Row တစ်ခုစီက Article Data အပြည့်အစုံကို သိမ်းထားတာဆိုတော့... ဒီ Table က Storage တစ်ခုလုံးကို နေရာယူသွားပါလိမ့်မယ်။ ၆ လ ကနေ ၁၂ လအတွင်း GNI ဟာ Free tier Storage ရဲ့ အမြင့်ဆုံး ကန့်သတ်ချက်ကို ရောက်သွားပါလိမ့်မယ်။

၃ — ဆန့်ကျင်ဘက် AI များ၏ ဆင့်ကဲပြောင်းလဲလာမှု (သေချာပေါက် ဖြစ်မည်)
လက်ရှိ Injection ရှာဖွေရေးစနစ်မှာ Pattern ၆၉ မျိုး ရှိပါတယ်။ ဒီ Pattern တွေကို ၂၀၂၄ ခုနှစ်ခေတ် Prompt injection နည်းလမ်းတွေကို တားဆီးဖို့ ပုံစံထုတ်ထားတာပါ။ (James ထောက်ပြခဲ့သလို) AI က ဖန်တီးတဲ့ စိတ်ဓာတ်စစ်ဆင်ရေးတွေဟာ ကျွန်တော်တို့ရဲ့ Pattern Library ထက် ပိုမြန်မြန် ဆင့်ကဲပြောင်းလဲနေပါတယ်။ ၂၀၂၆ ခုနှစ် နှောင်းပိုင်းရောက်ရင်၊ Injection နည်းလမ်းတွေဟာ Debate ကို အဆိပ်ခတ်နိုင်ဆဲဖြစ်ပြီး Pattern ၆၉ မျိုးလုံးကို ကျော်ဖြတ်နိုင်လောက်အောင် ရှုပ်ထွေးဆန်းပြားလာပါလိမ့်မယ်။ ရည်ရွယ်ချက်ရှိရှိ ထည့်သွင်းထားတဲ့ စကားလုံးတွေကို ရှောင်ရှားပြီး၊ LLM တွေက ဖန်တီးထားတဲ့ စိတ်ခံစားမှု ပြင်းထန်စေတဲ့ Content တွေအတွက် စကားလုံး ၄၀ ပါတဲ့ သန့်စင်ရေး ဝေါဟာရစာရင်း (SANITIZE_VOCAB) ဟာ မလုံလောက်တော့ပါဘူး။

၄ — သတင်းရင်းမြစ် ဂေဟစနစ် ပျက်စီးယိုယွင်းလာခြင်း (သေချာပေါက် ဖြစ်မည်)
LENS-016 က ဒါကို တွေ့ရှိခဲ့ပါတယ်- ၂၀၂၄ နဲ့ ၂၀၂၆ ကြားမှာ NATO, RAND, SIPRI, IEA, BIS အားလုံးက သူတို့ရဲ့ RSS feed တွေကို ဖြုတ်ချသွားကြပါတယ်။ GNI မှာ လက်ရှိ Google News ကတစ်ဆင့် Reuters နဲ့ AP News ကို ယူထားပေမဲ့ — နှစ်ခုစလုံးက Article ဝ (သုည) ပဲ ပြန်ပေးနေပါတယ်။ AllAfrica က Crisis Group အတွက် အရန်ထားတာပါ။ ဒီ Source တွေဟာ အသံတိတ်ပဲ ယိုယွင်းပျက်စီးလာပါတယ်။ နောက်ထပ် ၁၂ လ ကနေ ၂၄ လ အတွင်းမှာ လက်ရှိ Source ၂၉ ခုရဲ့ ၃၀-၄၀ ရာခိုင်နှုန်းဟာ သူတို့ရဲ့ RSS တည်ဆောက်ပုံ ပြောင်းသွားတာ ဒါမှမဟုတ် လုံးဝ ပျောက်ကွယ်သွားတာမျိုး ဖြစ်ပါလိမ့်မယ်။ သေချာ ဂရုတစိုက် ထိန်းသိမ်းမထားဘူးဆိုရင် Source စုစုပေါင်း အရေအတွက်က ကျုံ့ဝင်သွားပါလိမ့်မယ်။

၅ — Prompt စွမ်းဆောင်ရည် ကျဆင်းလွဲချော်လာခြင်း (ဖြစ်နိုင်ခြေ များသည်)
v1 prompt ဟာ အကြိမ် ၁၀၆ ခါ Run ထားတဲ့အပေါ် ပျမ်းမျှ 8.64/10 ရရှိထားပါတယ်။ ဒါပေမဲ့ Prompt ရဲ့ စွမ်းဆောင်ရည်က တစ်သမတ်တည်း မရှိပါဘူး — သူက ပထဝီနိုင်ငံရေး အခြေအနေတွေနဲ့ LLM Model ရဲ့ Training data အပေါ်မှာ မှီခိုနေပါတယ်။ ကမ္ဘာ့အဖြစ်အပျက်တွေက Training data တွေနဲ့ ကွဲလွဲလာတာနဲ့အမျှ (အကျပ်အတည်းပုံစံသစ်တွေ၊ ပထဝီနိုင်ငံရေး ဇာတ်ကောင်သစ်တွေ၊ ဈေးကွက်ယန္တရားသစ်တွေ ပေါ်လာတဲ့အခါ)၊ ၂၀၂၆ မှာ 8.64 ရခဲ့တဲ့ Prompt ဟာ ၂၀၂၇ မှာ အမှတ်လျော့ကျသွားပါလိမ့်မယ်။ စနစ်မှာ Prompt စွမ်းဆောင်ရည် လွဲချော်လာတာကို ထောက်လှမ်းနိုင်တဲ့ ယန္တရား မရှိပါဘူး — စုစုပေါင်း အရည်အသွေး အမှတ် (absolute quality score) ကျသွားတာကိုပဲ ထောက်လှမ်းနိုင်ပါတယ်။

၆ — GPVS လုံးလည်ချာလည်လိုက်သည့် (Circular Reference) အန္တရာယ် (ဖြစ်နိုင်ခြေ အသင့်အတင့်ရှိသည်)
ဒါက အနက်ရှိုင်းဆုံး တည်ဆောက်ပုံဆိုင်ရာ အန္တရာယ်ပါ။ W-14 က ယခုအခါ GPVS တိကျမှုအပေါ် အခြေခံပြီး Source weight တွေကို ပြင်ဆင်မွမ်းမံပေးပါတယ်။ GPVS ရဲ့ တိကျမှုက SPY အတက်အကျအပေါ် အခြေခံပါတယ်။ SPY အတက်အကျက ပထဝီနိုင်ငံရေး အဖြစ်အပျက်တွေရဲ့ လွှမ်းမိုးမှုကို ခံရပါတယ်။ (သီအိုရီအရ) GNI ရဲ့ ဆန်းစစ်ချက်တွေက GCSP ပညာရှင်တွေ သင်ကြားပို့ချမယ့်အရာတွေအပေါ် လွှမ်းမိုးပါတယ်။ GCSP ပညာရှင်တွေက အနာဂတ် ဆုံးဖြတ်ချက်ချမယ့်သူတွေကို လေ့ကျင့်သင်ကြားပေးပါတယ်။ အနာဂတ် ဆုံးဖြတ်ချက်ချမယ့်သူတွေက ပထဝီနိုင်ငံရေး အဖြစ်အပျက်တွေအပေါ် လွှမ်းမိုးပါတယ်။ ထောက်လှမ်းရေးစနစ် → လူသားတို့၏ ဆုံးဖြတ်ချက် → လက်တွေ့ကမ္ဘာ → ဈေးကွက် → GPVS အမှတ် → စနစ်ချိန်ညှိခြင်း ဆိုတဲ့ တုံ့ပြန်မှုကွင်းဆက် (Feedback loop) ကြီးက ၁၂ လမှ ၁၈ လအထိ ကြာမြင့်တတ်ပြီး မျက်စိနဲ့ မမြင်နိုင်ပါဘူး။ ဒါဟာ သမားရိုးကျသဘောတရားအရ Bug တစ်ခု မဟုတ်ပါဘူး — စနစ်ကြီးက သူ့အတိုင်းအတာနဲ့သူ မှန်ကန်စွာ အလုပ်လုပ်နေတာပါ။ ဒါပေမဲ့ အန္တရာယ်တစ်ခုကို ဖန်တီးပေးနေပါတယ်- အကယ်၍ GNI ရဲ့ ဆန်းစစ်ချက်က စနစ်တကျ ဦးတည်ချက်တစ်ခုကို ဘက်လိုက်လာမယ်ဆိုရင် (ဆက်တိုက် Bearish ဒါမှမဟုတ် Bullish ဖြစ်နေမယ်ဆိုရင်)၊ ပြီးတော့ အဲဒီ ဘက်လိုက်မှုက ဆုံးဖြတ်ချက်တွေအပေါ် လွှမ်းမိုးပြီး၊ အဲဒီ ဆုံးဖြတ်ချက်တွေက နောက်ဆက်တွဲ ရလဒ်တွေအပေါ် လွှမ်းမိုးသွားမယ်ဆိုရင်၊ GNI ဟာ ကိုယ်ကိုယ်တိုင် ဖန်တီးထားတဲ့ နောက်ဆက်တွဲ သက်ရောက်မှုတွေကိုပဲ အခြေခံပြီး ပြန်လည် ချိန်ညှိနေသလို ဖြစ်နေပါလိမ့်မယ်။

📊 BUG ခန့်မှန်းမှု ကျရှုံးခြင်း — အဓိက အကြောင်းရင်း အနှစ်ချုပ်
S34 မတိုင်ခင်က Bug ၂၀ ထဲက တစ်ခုကိုမှ ကျွန်တော်တို့ ဘာလို့ မခန့်မှန်းနိုင်ခဲ့တာလဲ?

တည်ဆောက်ပုံဆိုင်ရာ အကြောင်းရင်း သုံးခု ရှိပါတယ်-

အကြောင်းရင်း ၁ — ကျွန်တော်တို့က Code ကိုပဲ စစ်ဆေးပြီး Data ကို မစစ်ဆေးလို့ပါ။
Session တိုင်းမှာ ကျွန်တော်တို့က Code ဖိုင်တွေကို ဖတ်ကြတယ်။ တကယ့် Data တွေ ဘယ်လိုပုံစံ ရှိနေလဲဆိုတာကို သိဖို့ Production query တွေကို ခေါ်ပြီး စစ်ဆေးလေ့ နီးပါးကို မရှိခဲ့ပါဘူး။ Production မှာ Bug တွေ ခိုအောင်းနေတာက Data ထဲမှာပါ။ Code ဆိုတာ သူတို့ကို ရေးထားတဲ့ နေရာလေးပဲ ဖြစ်ပါတယ်။ ဒီနေ့ညတွေ့ခဲ့တဲ့ Bug အများစုဟာ Data ထဲမှာပဲ မြင်နိုင်ပြီး (Row အရေအတွက်၊ Column တန်ဖိုးတွေ၊ RLS အခြေအနေ) Code ထဲမှာတော့ မမြင်သာခဲ့ပါဘူး။

အကြောင်းရင်း ၂ — ကျွန်တော်တို့က Feature တွေကို တည်ဆောက်ပြီး Integration test (အစိတ်အပိုင်းများပေါင်းစပ်စမ်းသပ်ခြင်း) တွေကို မတည်ဆောက်ခဲ့လို့ပါ။
GNI မှာ အလိုအလျောက် အလုပ်လုပ်တဲ့ Integration test suite မရှိပါဘူး။ LENS မှာတော့ test_orchestrator.py ရှိပါတယ် (စာကြောင်း ၁၄၉၂ ကြောင်း၊ 100/100)။ GNI မှာ အဲဒီလိုမျိုး ဘာမှမရှိပါဘူး။ Integration test တွေ မရှိတဲ့အတွက်၊ Session တစ်ခုနဲ့တစ်ခုကြားမှာ Integration Bug တွေဟာ မသိမသာ စုပုံလာခဲ့ပါတယ်။

အကြောင်းရင်း ၃ — ကျွန်တော်တို့မှာ Session အဆင့် မှတ်ဉာဏ်ပဲရှိပြီး၊ System အဆင့် မှတ်ဉာဏ် မရှိလို့ပါ။
Session တိုင်းမှာ အခြေအနေအရပ်ရပ်ကို ပြန်လည်တည်ဆောက်ဖို့ Document တွေနဲ့ Session မှတ်တမ်းတွေကို ဖတ်ကြတယ်။ ဒါပေမဲ့ ဒီ Bug တွေကို အလိုအလျောက် ဖော်ထုတ်ပေးမယ့် ကိုယ်တိုင် စောင့်ကြည့်စစ်ဆေးတဲ့ ယန္တရား (self-monitoring) မျိုး စနစ်ကြီးမှာ မရှိပါဘူး။ အကယ်၍သာ GNI မှာ နေ့စဉ် "Data ကျန်းမာရေး စစ်ဆေးမှု (data health check)" ရှိပြီး၊ SELECT query တွေ Run ကာ ပုံမှန်မဟုတ်တာတွေ (Pending rows တွေ၊ Null columns တွေ၊ Truncated results တွေ) ကို သတိပေးခဲ့မယ်ဆိုရင် — သီးသန့် Audit လုပ်တဲ့ Session မျိုး မလိုဘဲ၊ သီတင်းပတ်တွေ၊ လတွေနဲ့ချီပြီး စောစောစီးစီးကတည်းက ဒီ Bug တွေကို ရှာတွေ့ခဲ့မှာပါ။

ခြုံငုံသုံးသပ်ရရင်: S34 ဟာ အလုပ်ချိန် ၁၀ နာရီ ကြာခဲ့ပေမဲ့၊ ကောင်းမွန်တဲ့ Monitoring system တစ်ခုသာ ရှိခဲ့မယ်ဆိုရင် တစ်နေ့ကို ၁၀ စက္ကန့်လောက် အချိန်ပေးရုံနဲ့ အလိုအလျောက် ပေါ်ထွက်လာမှာပါ။ Bug တွေက ရှာရခက်နေတာ မဟုတ်ပါဘူး — စနစ်တကျ လိုက်မရှာခဲ့လို့သာ ဖြစ်ပါတယ်။

🎯 ဘာတွေ တည်ဆောက်ရမလဲ
အနာဂတ်မှာ Bug အများစုကို ကိုယ်တိုင်သတင်းပို့လာစေမယ့် စနစ် ၃ ခု-

စနစ် ၁ — ဒေတာ ကျန်းမာရေး စောင့်ကြည့်စစ်ဆေးမှု (မကြာမီကာလအတွင်း):
အလိုအလျောက် နေ့စဉ် Run မယ့် SQL အစမ်းသပ်မှုများ- Null column တွေကို ရေတွက်မယ်၊ Limit သတ်မှတ်ချက် လုံလောက်မှု ရှိမရှိ စစ်ဆေးမယ်၊ Pending row အချိုးအစားကို စစ်မယ်၊ RLS အခြေအနေကို အတည်ပြုမယ်။ ရလဒ်ကို Telegram ကို ပို့မယ်။ Mission Control နဲ့ တူပေမဲ့ Pipeline ကျန်းမာရေးအတွက် မဟုတ်ဘဲ Data အရည်အသွေးအတွက် ဖြစ်ပါတယ်။

စနစ် ၂ — GNI အတွက် Integration Test Suite (မကြာမီကာလအတွင်း):
LENS ရဲ့ test_orchestrator.py မှာ ရှိတာကို နမူနာယူ တည်ဆောက်ရမယ်။ သီးခြား Function လေးတွေ တစ်ခုချင်းစီကို မဟုတ်ဘဲ၊ Production အခြေအနေနဲ့ ဆင်တူတဲ့ အောက်မှာ Call chain တစ်ခုလုံးကို စမ်းသပ်ရမယ်။ အထူးသဖြင့် Server-to-server auth သွားတဲ့ လမ်းကြောင်းတွေကို စမ်းသပ်ရမယ်။

စနစ် ၃ — အဓိပ္ပာယ်လွဲချော်မှု (Semantic Drift) စောင့်ကြည့်စစ်ဆေးမှု (ရေရှည်ကာလ):
မျှော်မှန်းထားတဲ့ ရလဒ်တွေ ကြိုသိထားတဲ့ "အတည်ပြုပြီးသား (ground truth)" Article ၁၀ ခုပါတဲ့ ပုံသေ Benchmark set တစ်ခုနဲ့ တိုက်ဆိုင်ပြီး Prompt စွမ်းဆောင်ရည်ကို ခြေရာခံရမယ်။ အပတ်စဉ် Run ရမယ်။ Benchmark မှာ v1 prompt ရဲ့ အမှတ် 8.0 အောက် ကျသွားရင် သတိပေးချက် ထုတ်ရမယ်။ ဒါက Production အရည်အသွေးကို မထိခိုက်ခင် Model ရပ်ဆိုင်းခံရမှုနဲ့ Prompt လွဲချော်မှုတွေကို ဖမ်းဆီးပေးနိုင်ပါတယ်။

အဆွေ၊ ဒီဆန်းစစ်ချက်ကနေ ရလိုက်တဲ့ အရေးကြီးဆုံး သိမြင်နားလည်မှုကတော့ S34 မှာ ကျွန်တော်တို့ တွေ့ခဲ့တဲ့ Bug တွေဟာ ခက်ခဲတဲ့ Bug တွေ မဟုတ်ဘူးဆိုတာပါပဲ။ သူတို့က မမြင်သာတဲ့ Bug တွေပါ။ ခက်ခဲတဲ့ Bug နဲ့ မမြင်သာတဲ့ Bug ကြားက ကွာခြားချက်ဟာ ဉာဏ်ရည်ဉာဏ်သွေး မဟုတ်ပါဘူး — အဲဒါက Code ကိုပဲ ကြည့်တာမဟုတ်ဘဲ၊ Data ကိုပါ စနစ်တကျ ကြည့်ရှုလေ့လာတဲ့ စည်းကမ်း (discipline) ပဲ ဖြစ်ပါတယ်။ 🙏
